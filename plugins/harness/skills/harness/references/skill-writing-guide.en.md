## 1. Description Writing Patterns

A description is the skill's only trigger mechanism. Claude decides whether to use a skill based solely on its name + description in the `available_skills` list.

### Understanding the Trigger Mechanism

Claude tends not to invoke a skill for simple tasks it can handle with its built-in tools. A simple "read this PDF" may not trigger even with a perfect description. Complex, multi-step, specialized tasks are far more likely to trigger a skill.

### Writing Principles

1. State **what the skill does** + **concrete trigger situations**
2. Specify boundary conditions that distinguish cases that look similar but should NOT trigger
3. Be slightly "pushy" — compensate for Claude's conservative trigger judgment

### Good Examples

```yaml
description: "Reads PDF files; extracts text/tables; merges, splits, rotates,
  watermarks, encrypts/decrypts, OCRs — all PDF work. When the user mentions
  a .pdf file or requests a PDF artifact, use this skill. Especially useful
  for conversion/editing/analysis, not for mere 'read it' requests."
```

```yaml
description: "All spreadsheet work for xlsx/csv/tsv files: column add,
  formulas, formatting, charts, data cleaning. When the user mentions a
  spreadsheet file — even casually ('the xlsx in Downloads') — use this skill."
```

### Bad Examples

- `"A skill that processes data"` — too vague; what files / what work?
- `"PDF-related work"` — no concrete actions; trigger situations missing

## 2. Body Writing Style

### Why-First Principle

When the LLM understands the reason, it judges edge cases correctly. Conveying context beats coercive rules.

**Bad:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good:**
```markdown
Use pdfplumber for table extraction. PyPDF2 specializes in text extraction
and does not preserve the row/column structure of tables. pdfplumber
recognizes cell boundaries and returns structured data.
```

### Generalization Principle

When feedback or test results reveal a problem, **generalize at the principle level** rather than applying a narrow fix tied to a specific example.

**Overfitted fix:**
```markdown
If a "Q4 revenue" column exists, convert it to numeric.
```

**Generalized fix:**
```markdown
If a column name contains numeric-implying keywords ("revenue", "amount",
"quantity"), convert the column to numeric. On conversion failure, keep
the original value.
```

### Imperative Tone

Use "do X" / "X is …" forms, not "you may" / "you can". A skill is an instruction sheet.

### Save Context

The context window is a common resource. Ask whether each sentence earns its token cost:
- "Does Claude already know this?" → delete
- "Without this, does Claude make a mistake?" → keep
- "Would one concrete example beat a long explanation?" → replace with the example

## 3. Output-Format Definition Patterns

Use when artifact format matters:

```markdown
## Report Structure
Follow this template exactly:

# [Title]
## Summary
## Key Findings
## Recommendations
```

Keep format definitions terse; including a real example is more effective.

## 4. Example Writing Patterns

Examples beat long explanations:

```markdown
## Commit Message Format

**Example 1:**
Input: Add JWT-token-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: Fix bug where the password-reveal button does not work on the login page
Output: fix(login): repair password-reveal toggle button
```

## 5. Progressive Disclosure Patterns

### Pattern 1: Per-Domain Split

```
bigquery-skill/
├── SKILL.md (overview + domain-selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)
```

When the user asks about revenue, only `finance.md` is loaded.

### Pattern 2: Conditional Detail

```markdown
# DOCX Processing

## Document Generation
Use docx-js to create a new document. → see [DOCX-JS.md](references/docx-js.md).

## Document Editing
For simple edits, modify the XML directly.
**If you need tracked changes**: see [REDLINING.md](references/redlining.md)
```

### Pattern 3: Large Reference File Structure

Reference files longer than 300 lines should include a table of contents at the top:

```markdown
# API Reference

## Contents
1. [Authentication](#authentication)
2. [Endpoints](#endpoints)
3. [Error Codes](#error-codes)
4. [Rate Limits](#rate-limits)

---

## Authentication
...
```

## 6. Script Bundling Criteria

Observe agents' transcripts during test runs. Bundle when you see these signals:

| Signal | Action |
|--------|--------|
| Same helper script generated in 3 of 3 tests | Bundle under `scripts/` |
| Same `pip install` / `npm install` run every time | Specify dependency-install step in the skill |
| Same multi-step approach repeated | Document as the standard procedure in the skill body |
| Same workaround applied after a similar error each time | Record the known issue and its fix in the skill |

Bundled scripts must be run through execution tests.

## 7. Data Schema Standards

Use standard schemas for consistent inter-skill data exchange. Useful for testing/eval of skills the harness generates.

### eval_metadata.json

Metadata per test case:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "The user's task prompt",
  "assertions": [
    "Artifact contains X",
    "File created in Y format"
  ]
}
```

### grading.json

Assertion-based grading result:

```json
{
  "expectations": [
    {
      "text": "Artifact contains 'Seoul'",
      "passed": true,
      "evidence": "Step 3 confirms 'Seoul-region data extraction'"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field name caution:** use `text`, `passed`, `evidence` exactly (no variants like `name` / `met` / `details`).

### timing.json

Execution time / token measurement:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Save `total_tokens` and `duration_ms` from the sub-agent completion notification immediately. The data is accessible only at notification time and cannot be recovered later.

## 8. What Not To Include in a Skill

- Auxiliary docs like README.md, CHANGELOG.md, INSTALLATION_GUIDE.md
- Meta-info from the skill-creation process (test results, iteration history)
- User-facing manuals (a skill is an instruction sheet for the AI agent)
- General knowledge Claude already has
