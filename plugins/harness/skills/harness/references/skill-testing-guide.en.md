# Skill Testing & Iterative Improvement Guide

> **Read at phase:** Phase 6 (self-verification after skill generation) + Phase 8 (trigger regression check). Logs the should / should-NOT trigger matrix.

Methodology for validating and iteratively improving the quality of skills generated by the harness. Supplementary reference for SKILL.md Phase 6·8.

> **Heuristic threshold catalog + tuning guide (P2-4 — 2026-05-14):** Thresholds in this doc are *experience-based* heuristics. Adjust by domain complexity.
>
> | Threshold | Default | Rationale | Tuning Recommendation |
> |---|---|---|---|
> | Initial test prompts (§2) | 2–3 | Heuristic — covers core + edge + (optional) composite | Complex domains (ML / domain-specific DSL): 5–7 |
> | Test-prompt coverage categories (§2) | Core use / edge / composite (optional) | Heuristic — isolates common failure modes | Safety / regulated domains: add *negative cases* (verify rejection of invalid input) |
> | Trigger regression eval queries (§7) | 20 (should-trigger 10 + should-NOT 10) | Heuristic — 8–10 is the floor, 10+10 absorbs domain ambiguity | Simple domain (CRUD): 8+8 / ambiguous domain (ML "model" overload): 15+15 |
> | should-trigger / should-NOT ratio (§7) | 1:1 (10:10) | Heuristic — symmetric false-positive ↔ false-negative | Recall-first domain (security detection): 12:8 / precision-first (cost-sensitive): 8:12 |
> | Train/Test split (§7) | 60% / 40% | Heuristic — 12 Train for description tuning + 8 Test for regression | ≥30 queries → 70/30 for statistical stability |
> | Near-miss ratio (§7) | Concentrate on the should-NOT side | Heuristic — boundary ambiguity is the trigger-accuracy crux | Domain jargon conflicts (ML "model" / modeling): ≥50% near-miss |
> | Trigger accuracy floor (§7) | <8 queries → quality not guaranteed | Heuristic — variance accumulation floor | 8 is the absolute floor; 10 is the default recommendation |
>
> This table is a *baseline* — adjust for domain complexity and jargon-conflict frequency. When changing thresholds, also update `_baseline/intent_profile.md`'s `quality.test_rigor` field.

## 1. Testing Framework Overview

Skill quality validation is a combination of **qualitative** and **quantitative** evaluation.

| Type | Method | Suited Skills |
|------|--------|---------------|
| **Qualitative** | User directly reviews artifacts | Style, design, creative work — subjective quality |
| **Quantitative** | Assertion-based automated grading | File generation, data extraction, code generation — objectively verifiable |

Core loop: **write → run test → evaluate → improve → retest**.

## 2. Writing Test Prompts

### Principle

Test prompts should be **concrete and natural sentences a real user might type**. Abstract or artificial prompts have low test value.

### Bad Examples

```
"Process the PDF"
"Extract the data"
"Generate a chart"
```

### Good Examples

```
"In the 'Q4_revenue_final_v2.xlsx' in my Downloads folder, use column C (revenue)
and column D (cost) to add a profit-margin (%) column. Then sort by margin descending."
```

```
"Extract the page-3 table from this PDF and convert to CSV. The header has two
rows — first is the category, second is the actual column names."
```

### Prompt Diversity

- Mix **formal / casual** tone
- Mix **explicit / implicit** intent (file format stated directly vs. inferred from context)
- Mix **simple / complex** tasks
- Include abbreviations, typos, and casual phrasing in a few

### Coverage

Start with 2–3 prompts that together cover:
- 1 core use case
- 1 edge case
- (Optional) 1 composite task

## 3. Execution Test: With-skill vs Baseline

### 3-1. Comparative Run Structure

For each test prompt, spawn two sub-agents **concurrently**:

**With-skill run:**
```
prompt: "{test prompt}"
skill_path: {skill_path}
output_path: _workspace/iteration-N/eval-{id}/with_skill/outputs/
```

**Baseline run:**
```
prompt: "{test prompt}"   (identical)
skill: none
output_path: _workspace/iteration-N/eval-{id}/without_skill/outputs/
```

### 3-2. Baseline Selection

| Situation | Baseline |
|-----------|----------|
| New skill | Same prompt without the skill |
| Improving existing skill | Pre-edit skill version (snapshot preserved) |

### 3-3. Capture Timing Data

Save `total_tokens` and `duration_ms` from the sub-agent completion notification **immediately**. The data is accessible only at notification time and cannot be recovered later.

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

## 4. Quantitative Evaluation: Assertion-Based Grading

### 4-1. Writing Assertions

When artifacts are objectively verifiable, define assertions for automated grading.

**Good assertion:**
- Pass/fail determinable objectively
- Descriptive name — the result alone shows what is being checked
- Verifies the skill's core value

**Bad assertion:**
- Passes regardless of the skill (e.g., "output exists")
- Requires subjective judgment (e.g., "well written")

### 4-2. Programmable Checks

If an assertion can be verified by code, script it. Faster and more reliable than manual inspection, and reusable across iterations.

### 4-3. Beware Non-discriminating Assertions

An assertion that passes "100% in both configurations" cannot measure the skill's differential value. Remove or replace such assertions with more demanding ones.

### 4-4. Grading Result Schema

```json
{
  "expectations": [
    {
      "text": "Profit-margin column was added",
      "passed": true,
      "evidence": "Column 'profit_margin_pct' present in column E"
    },
    {
      "text": "Sorted by margin descending",
      "passed": false,
      "evidence": "No sort applied — original order retained"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}
```

## 5. Specialist Agents

Using specialist agents during testing/evaluation raises quality.

### 5-1. Grader

Performs assertion-based grading and extracts verifiable claims from artifacts for cross-validation.

**Role:**
- Pass/fail per assertion + evidence
- Extract factual claims from artifacts and verify
- Feedback on eval quality itself (suggest changes when an assertion is too easy or ambiguous)

### 5-2. Comparator (Blind A/B)

Anonymize the two artifacts as A/B and judge quality without knowing which used the skill.

**When to use:** When you want a rigorous answer to "Is the new version really better?". Often omitted in routine iteration.

**Judging criteria:**
- Content: accuracy, completeness
- Structure: organization, formatting, usability
- Overall score

### 5-3. Analyzer

Analyzes statistical patterns in benchmark data:
- Non-discriminating assertions (pass in both → no differentiation)
- High-variance evals (results vary across runs → unstable)
- Time/token trade-offs (skill raises quality but also raises cost)

## 6. Iterative Improvement Loop

### 6-1. Collect Feedback

Show artifacts to the user and collect feedback. Treat empty feedback as "no issue".

### 6-2. Improvement Principles

1. **Generalize feedback** — narrow fixes tied to a test example are overfitting. Fix at the principle level.
2. **Cut what does not earn its weight** — read transcripts; if the skill makes the agent do unproductive work, delete that section.
3. **Explain the Why** — even when user feedback is terse, understand why it matters and embed that understanding into the skill.
4. **Bundle repeated work** — if every test run generates the same helper script, ship it under `scripts/`.

### 6-3. Iteration Procedure

```
1. Edit the skill
2. Rerun all test cases under a new iteration-N+1/ directory
3. Present results to the user (compare with previous iteration)
4. Collect feedback
5. Edit again → repeat
```

**Stop when:**
- The user is satisfied
- All feedback is empty (no issues anywhere)
- No meaningful improvement remains

### 6-4. Draft → Review Pattern

When editing the skill, write a draft and then **re-read with fresh eyes** to improve it. Do not try to write perfectly on the first pass — go through a draft-review cycle.

## 7. Description Trigger Verification

### 7-1. Writing Trigger-Eval Queries

Write 20 eval queries — 10 should-trigger + 10 should-NOT-trigger.

**Query quality criteria:**
- Concrete, natural sentences a real user would type
- Include specific details: file paths, personal context, column names, company names
- Mix length, tone, and format
- Focus on **edge cases** rather than obvious answers

**Should-trigger (8–10):**
- The same intent expressed differently (formal / casual)
- Cases that clearly need the skill without explicitly naming it / the file type
- Non-mainstream uses
- Cases that compete with other skills but should still go to this one

**Should-NOT-trigger (8–10):**
- **Near-misses are the point** — queries that share keywords but should go to a different tool/skill
- Obviously unrelated queries ("write a Fibonacci function") have no test value
- Adjacent domains, ambiguous phrasings, keyword overlap with different context

### 7-2. Conflict Check With Existing Skills

Verify the new skill's description does not overlap the trigger area of existing skills:

1. Collect descriptions of existing skills
2. Confirm the new skill's should-trigger queries do not mis-trigger existing skills
3. On conflict, sharpen the boundary conditions in the description

### 7-3. Automated Optimization (Optional Advanced)

When description optimization is needed:

1. Split the 20 eval queries Train(60%) / Test(40%)
2. Measure trigger accuracy with the current description
3. Analyze failures and produce an improved description
4. Select the best description on the Test set (not Train — avoids overfitting)
5. Up to 5 iterations

> Run this as an automation script using `claude -p`. Token cost is high — run as a final step once the skill is sufficiently stable.

## 8. Workspace Layout

A directory layout for managing test/eval results systematically:

```
{skill-name}-workspace/
├── iteration-1/
│   ├── eval-descriptive-name-1/
│   │   ├── eval_metadata.json
│   │   ├── with_skill/
│   │   │   ├── outputs/
│   │   │   ├── timing.json
│   │   │   └── grading.json
│   │   └── without_skill/
│   │       ├── outputs/
│   │       ├── timing.json
│   │       └── grading.json
│   ├── eval-descriptive-name-2/
│   │   └── ...
│   └── benchmark.json
├── iteration-2/
│   └── ...
└── evals/
    └── evals.json
```

**Rules:**
- Use **descriptive names** rather than numbers for eval directories (e.g., `eval-multi-page-table-extraction`)
- Each iteration is preserved in its own directory (never overwrite previous iterations)
- Do not delete `_workspace/` — it is used for post-hoc verification and audit trails.
